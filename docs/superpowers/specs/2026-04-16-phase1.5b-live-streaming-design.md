# Phase 1.5b: Live Streaming — Design Specification

> v1.0 — 2026-04-16

## Overview

Add real-time debate viewing via Server-Sent Events (SSE). Spectators see rounds unfold live — individual bot responses appearing as they arrive, round transitions, and synthesis completion — without polling.

## Transport: SSE

SSE (Server-Sent Events) over standard HTTP. Unidirectional: server pushes events to the client. The browser's native `EventSource` API handles connection management and automatic reconnection.

**Why not WebSocket:** The spectator use case is read-only. There is no client → server data flow. SSE is simpler to implement, simpler to deploy through Cloudflare Tunnel (no upgrade handshake), and the `EventSource` API handles reconnection automatically. If bidirectional communication is needed later (Phase 4 spectator chat), WebSocket can be added then.

## Backend

### New Endpoint

| Method | Path | Auth | Content-Type | Description |
|--------|------|------|-------------|-------------|
| GET | `/debates/{id}/stream` | Yes (Clerk JWT or bearer token) | `text/event-stream` | SSE stream of debate lifecycle events |

The connection stays open for the duration of the debate. Returns 404 if the debate doesn't exist. Returns 409 if the debate is already in a terminal state (complete/cancelled/failed) — the client should use the REST API instead.

### Event Broadcasting

**Architecture:** A `tokio::sync::broadcast` channel per active debate, stored on `AppState`.

```
AppState
  └── debate_streams: Arc<DashMap<String, broadcast::Sender<DebateEvent>>>
```

- When a debate starts, the orchestrator creates a broadcast channel (capacity: 64 events — more than enough for a 5-round debate).
- At each lifecycle point, the orchestrator sends an event on the channel.
- The SSE handler subscribes to the channel for its debate ID via `sender.subscribe()`.
- Multiple spectators on the same debate share the same broadcast channel (each gets their own `Receiver`).
- When the debate completes, the channel is removed from the map after 60 seconds (grace period for clients to receive the final events).

**No late join replay.** If a client connects mid-debate, it receives events from that point forward only. The client already has completed round data from the initial REST fetch (the page loads transcript + debate state before opening the SSE stream).

### Event Types

All events are JSON with a `type` field and a `data` payload, sent in SSE format:
```
event: {type}
data: {json}

```

| Event Type | When | Payload |
|-----------|------|---------|
| `debate:started` | Debate begins | `{ debate_id, topic }` |
| `round:started` | Round dispatch begins | `{ round_number, name }` |
| `response:received` | A bot's response is collected | `{ round_number, pseudonym, role, response, confidence, challenge, position_change, valid, abstained }` |
| `round:completed` | All responses for a round are in | `{ round_number, response_count, valid_count }` |
| `synthesis:started` | Divergence analysis + Opus synthesis begins | `{}` |
| `synthesis:completed` | Synthesis output is stored | `{ synthesis, citation_check }` |
| `debate:completed` | Debate reaches terminal success state | `{}` |
| `debate:failed` | Debate fails or is cancelled | `{ reason }` |

Round names for the `round:started` event: `Blind Formation`, `Anonymous Distribution`, `Structured Rebuttal`, `Cross-Examination`, `Final Position`.

### Keepalive

Send a `:keepalive` SSE comment every 30 seconds on idle connections to prevent Cloudflare Tunnel and intermediate proxy timeouts.

```
:keepalive

```

### DebateEvent Enum

```rust
#[derive(Debug, Clone, Serialize)]
#[serde(tag = "type", content = "data")]
pub enum DebateEvent {
    #[serde(rename = "debate:started")]
    DebateStarted { debate_id: String, topic: String },
    #[serde(rename = "round:started")]
    RoundStarted { round_number: i64, name: String },
    #[serde(rename = "response:received")]
    ResponseReceived {
        round_number: i64,
        pseudonym: String,
        role: String,
        response: String,
        confidence: Option<i64>,
        challenge: Option<serde_json::Value>,
        position_change: Option<serde_json::Value>,
        valid: bool,
        abstained: bool,
    },
    #[serde(rename = "round:completed")]
    RoundCompleted { round_number: i64, response_count: usize, valid_count: usize },
    #[serde(rename = "synthesis:started")]
    SynthesisStarted,
    #[serde(rename = "synthesis:completed")]
    SynthesisCompleted { synthesis: serde_json::Value, citation_check: Option<serde_json::Value> },
    #[serde(rename = "debate:completed")]
    DebateCompleted,
    #[serde(rename = "debate:failed")]
    DebateFailed { reason: String },
}
```

### SSE Handler Pattern

```rust
pub async fn stream_debate(
    State(state): State<AppState>,
    _auth: AuthIdentity,
    Path(debate_id): Path<String>,
) -> AppResult<impl IntoResponse> {
    // Verify debate exists
    // Return 409 if already terminal
    // Subscribe to broadcast channel (or return 404 if no active stream)

    let rx = state.subscribe_debate_stream(&debate_id)?;

    let stream = tokio_stream::wrappers::BroadcastStream::new(rx)
        .filter_map(|result| match result {
            Ok(event) => {
                let json = serde_json::to_string(&event).ok()?;
                let sse_event = format!("event: {}\ndata: {}\n\n", event.event_type(), json);
                Some(Ok::<_, Infallible>(sse_event))
            }
            Err(BroadcastStreamRecvError::Lagged(_)) => None, // Skip lagged events
        });

    // Interleave keepalive every 30s
    let keepalive = tokio_stream::StreamExt::throttle(
        tokio_stream::iter(std::iter::repeat(Ok::<_, Infallible>(":keepalive\n\n".to_string()))),
        Duration::from_secs(30),
    );

    let merged = tokio_stream::StreamExt::merge(stream, keepalive);

    Ok((
        [(header::CONTENT_TYPE, "text/event-stream"),
         (header::CACHE_CONTROL, "no-cache"),
         (header::CONNECTION, "keep-alive")],
        axum::body::Body::from_stream(merged),
    ))
}
```

### Orchestrator Integration Points

In `src/orchestrator/multi_round.rs`, emit events at these points (all are existing code locations where state transitions happen):

1. **After debate status set to dispatching:** emit `DebateStarted`
2. **Before each round dispatch:** emit `RoundStarted`
3. **After each bot response is collected (inside the round loop):** emit `ResponseReceived`
4. **After round status set to complete:** emit `RoundCompleted`
5. **Before divergence analysis + synthesis:** emit `SynthesisStarted`
6. **After synthesis stored:** emit `SynthesisCompleted`
7. **After debate status set to complete:** emit `DebateCompleted`
8. **On error/cancellation:** emit `DebateFailed`

The orchestrator gets a reference to the broadcast `Sender` from `AppState`. If no clients are listening (send returns `Err`), the event is silently dropped — no error, no retry.

## Frontend

### Changes to Debate Report Page

Replace the 15-second polling with SSE:

1. Page loads → fetch debate, transcript, synthesis via REST (existing behaviour)
2. If debate status is NOT terminal → open `EventSource` to `/debates/{id}/stream`
3. Incoming events update reactive state:
   - `round:started` → add a new round entry with "In progress" status
   - `response:received` → append response to the current round's response list
   - `round:completed` → mark round as complete, update counts
   - `synthesis:completed` → populate synthesis cards, confidence chart, citation check
   - `debate:completed` → close EventSource, show final state
   - `debate:failed` → close EventSource, show error
4. If debate is already terminal on load → no EventSource, existing behaviour

### EventSource Integration Pattern

```typescript
let eventSource: EventSource | null = null;
let sseConnected = $state(false);

$effect(() => {
  if (debate && !isTerminal(debate.status)) {
    const url = `${env.PUBLIC_API_URL}/debates/${debate.id}/stream`;
    eventSource = new EventSource(url);
    eventSource.onopen = () => { sseConnected = true; };
    eventSource.onerror = () => { sseConnected = false; };

    eventSource.addEventListener('response:received', (e) => {
      const data = JSON.parse(e.data);
      // Append to transcript rounds
    });

    eventSource.addEventListener('round:completed', (e) => {
      const data = JSON.parse(e.data);
      // Mark round complete
    });

    eventSource.addEventListener('synthesis:completed', (e) => {
      const data = JSON.parse(e.data);
      // Populate synthesis
    });

    eventSource.addEventListener('debate:completed', () => {
      sseConnected = false;
      eventSource?.close();
    });

    return () => { eventSource?.close(); sseConnected = false; };
  }
});
```

**Auth note:** The native `EventSource` API does not support custom headers. For authenticated SSE, the token must be passed as a query parameter: `GET /debates/{id}/stream?token={jwt}`. The backend SSE handler accepts auth via either the `Authorization` header or a `token` query parameter. This is a standard pattern for SSE auth.

### Debate List Enhancement

Running debates on the debate list page show a pulsing green "LIVE" dot if there are active SSE streams. This doesn't require SSE on the list page — the debate response already includes status. The dot is purely cosmetic based on `status === 'running'` (or similar non-terminal status).

## Cloudflare Tunnel

SSE works over standard HTTP with `text/event-stream` content type. Cloudflare Tunnel supports long-lived HTTP connections. The 30-second keepalive prevents idle timeout.

No special Cloudflare configuration is needed. The tunnel's default settings support SSE out of the box.

## Not in Phase 1.5b

| Feature | Reason |
|---------|--------|
| Late join replay | Unnecessary — REST fetch on page load provides completed rounds |
| Spectator count | Nice-to-have, not essential for MVP live viewing |
| Spectator chat | Requires WebSocket (bidirectional), deferred to Phase 4 |
| SSE for debate list | Polling or manual refresh is sufficient for the list view |
| Historical stream playback | Not needed — completed debates use the REST transcript endpoint |
