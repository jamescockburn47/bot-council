# Ship's Log (Operator Legibility Phase 1) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Phase 1 of `docs/superpowers/specs/2026-07-02-operator-legibility-design.md`: the `system_events` journal with an authored narrative catalogue, writers at pool-available sites, an admin events API, `docs/RUNBOOK.md`, and a CI grep that stops runbook commands rotting.

**Architecture:** `src/observability/events.rs` owns the table access (record + query, fire-and-forget on write failure); `src/observability/system_guidance.rs` owns the authored catalogue (kind → severity, narrative template, suggested action) mirroring `error_guidance.rs`. Writers are single calls at sites that already hold the pool: boot (`lib.rs`), debate failure/quorum/synthesis-store (`multi_round.rs`), resynth CLI. Narratives are deterministic templates — never model-generated (spec's binding boundary).

**Tech Stack:** Rust 2024, sqlx/SQLite, axum. All cargo on EVO via `./scripts/sync-evo.sh`. Branch `claude/ships-log` off origin/main.

**Phase 1 event kinds (writers in this PR):** `service_started`, `model_route_changed`, `debate_failed`, `quorum_not_met`, `synthesis_fallback`, `sentinel_violation` (artifact-level, checked post-store in multi_round where the pool lives), `resynth_run`. Extraction-level sentinel events and bot-scoped events land with lifecycle Phase 4 (no pool at those seams; noted in the catalogue docs).

`events.rs` API (complete signatures):

```rust
pub struct EventScope<'a> { pub label: &'a str, pub debate_id: Option<&'a str>, pub bot_id: Option<&'a str> }
/// Fire-and-forget: looks up the catalogue, inserts, and on ANY failure
/// logs at warn and returns — the journal never breaks what it narrates.
pub async fn record_event(pool: &SqlitePool, kind: &str, scope: EventScope<'_>, detail: &str, technical_detail: Option<serde_json::Value>)
#[derive(sqlx::FromRow, serde::Serialize)] pub struct SystemEventRow { id, created_at, severity, event_kind, narrative, suggested_action, technical_detail, debate_id, bot_id }
pub async fn recent_events(pool, limit: i64, severity: Option<&str>) -> Result<Vec<SystemEventRow>, sqlx::Error>
```

Tests (in-file): record + fetch round-trip on in-memory pool; unknown kind still records (fallback narrative); poisoned pool (closed) → `record_event` returns without error; severity filter works.

### Task 2: The catalogue — `src/observability/system_guidance.rs`

Mirrors `error_guidance.rs`. Complete API:

```rust
pub struct EventTemplate { pub severity: &'static str, pub narrative: String, pub suggested_action: Option<&'static str> }
pub const KNOWN_EVENT_KINDS: [&str; 7] = [ ...the seven kinds... ];
/// scope_label is a short human handle ("debate 1a2b3c4d", "the analysis route");
/// detail is one plain-English clause composed by the writer.
pub fn compose(kind: &str, scope_label: &str, detail: &str) -> EventTemplate
```

Narratives per spec Part 1's table, e.g. `synthesis_fallback` → severity `attention`, narrative `format!("The summary for {scope_label} couldn't be fully structured, so a simplified version was stored. The debate itself is intact. {detail}")`, action `Some("Re-running the summariser usually fixes this: see RUNBOOK.md §5 (rebuild summaries).")`. `sentinel_violation` narrative names the sentinel ID from `detail` and says "Nothing was blocked; the item is flagged." Unknown kind → severity `attention`, narrative marks it unrecognised (and the closed-set test catches it in CI).

Tests: every KNOWN kind yields non-empty narrative; `problem`/`attention` kinds carry an action; unknown falls back; severities are within the closed set.
